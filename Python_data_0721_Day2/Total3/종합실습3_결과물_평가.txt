종합실습 3 - 결과물 평가

- guide의 step0~step7 코드를 하나씩 따라가며 설정(Config) → 집계(aggregate) → Jinja2 템플릿 → HTML 렌더링(render) → 루프/schedule/cron 세 가지 실행 방식 → run_once()로 이어지는 전체 흐름을 관심사별로 분리해 capstone03_automation/config.py·report.py·run_scheduler.py로, 그리고 한 파일로 합친 capstone03_automation.py로 작성
- 성공 판정 기준(report.py 실행 시 output/에 타임스탬프 HTML 생성, 브라우저에서 KPI·카테고리별 매출표 렌더링, --interval 반복 실행/Ctrl+C 중지, 루프·schedule·cron 결과 동일)에 맞춰 실제로 report.py 단발 실행, run_scheduler.py를 --interval 2~5초로 짧게 돌려 반복 생성 확인, --mode schedule로 같은 테스트를 한 번 더 돌려 두 방식이 동일하게 동작하는 것까지 확인

[잘한 점]
- 성공 기준을 말로만 확인하지 않고 report.py 단발 실행 → output/ 파일 생성 확인 → 브라우저로 열어 KPI·표 렌더링 스크린샷까지, 그리고 run_scheduler.py를 loop/schedule 두 모드로 각각 짧은 간격으로 실행해 로그가 반복 찍히고 Ctrl+C(타임아웃)로 정상 종료되는 것까지 단계별로 직접 실행해서 검증했다.
- loop 방식(run_scheduler.py --interval)과 schedule 라이브러리 방식을 별도 파일로 중복 작성하지 않고 --mode 옵션 하나로 묶어서 둘 다 결국 같은 run_once()만 호출하게 만들어, "실행 방식이 달라도 결과는 동일해야 한다"는 일관성 조건을 구조적으로 보장했다.
- config.py의 경로를 스크립트 위치 기준 절대경로(BASE_DIR)로 잡아서, capstone03_automation.py(같은 폴더에서 실행)와 capstone03_automation/report.py(하위 폴더에서 실행)를 어느 위치에서 실행해도 같은 output/ 폴더에 같은 결과가 쌓이도록 만들었다.

[어려웠던 점 / 개선할 점]
- guide의 CONFIG.data_path 기본값(data/sales_raw.csv)이 amount·category 컬럼이 있는 데이터를 전제로 하는데, 리포지토리 안에는 그런 파일이 없었다. practice4/sales_raw.csv(quantity·unit_price·discount)로 amount를 직접 계산해 대체 데이터를 만들었는데, 실제 매출 데이터가 아니라 실습용으로 파생시킨 값이라는 한계가 있다.
- run_once()가 실행될 때마다 output/에 report_*.html이 새로 쌓이기만 하고 오래된 리포트를 지우거나 개수를 제한하는 로직이 없다. 짧게 테스트할 때는 파일 몇 개로 끝나지만, 60초 주기로 하루 이상 돌리면 output/ 폴더에 리포트가 계속 늘어난다는 점을 생각해 이를 개선하는 코드가 필요함을 느꼈다.
