실습 2 - 결과물 평가

- Pydantic v2의 BaseModel로 User(바깥 모델)와 Profile(안쪽 모델)을 나눠 만들고, User 안에 profile: Profile 필드를 두어 중첩 구조를 검증
- Field(ge, le)로 age(0~120), profile.score(0~100)의 값 범위를 제한
- api_response.json의 40건을 한 건씩 User(**row)로 검증해 valid/invalid 리스트로 분리하고, ValidationError 발생 시 e.errors()로 사유를 모아 저장
- invalid 건에 대해 행/필드/사유 표로 출력 → 실행 결과 전체 40건 / 유효 36건 / 오염 4건으로 결과가 일치함을 확인

[잘한 점]
- 주어진 guide의 step0~step6 코드를 파일 하나하나 따라 작성하고 읽어보면서, 각 라인에 주석을 달아가며 이해한 뒤 다음 단계로 넘어가는 방식으로 실습을 진행해서 최종 실습1 코드의 전체 흐름을 수월하게 따라갈 수 있었다.
- Field만으로는 표현이 안 되는 이메일 형식 검사를 field_validator + classmethod 조합으로 구현해보며, 왜 field_validator는 cls를 받아야 하는지까지 실습하면서 이해할 수 있었다.
- Profile을 User 안에 필드 타입으로 그대로 넣어서 중첩 모델을 구성한 부분이 두 단계로 겹친 데이터도 자동으로 검증되는 걸 확인할 수 있었다.

[아쉬운 점 / 어려웠던 점]
- email 검증을 '@'와 '.' 유무만으로 판단하여, 형식은 맞지만 실제로는 이상한 값의 이메일까지는 걸러내지 못하는 한계가 있을 것 같아 보완하면 좋을 것 같다.
- ValidationError가 필드 하나만 틀려도 바로 멈추는 게 아니라, 한 레코드 안에서 여러 필드가 동시에 잘못됐을 때 그 에러들을 전부 모아서 e.errors()로 한 번에 리스트로 돌려준다는 점 이해가 어려웠다.
- ser(**row) 처럼 딕셔너리를 **로 언패킹해서 함수 인자로 넘기면, row의 각 key가 User 필드 이름과 자동으로 매칭돼서 각각 id=..., username=..., profile={...} 형태로 전달된다는 동작 방식을 처음엔 이해가 어려웠다.
